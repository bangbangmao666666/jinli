# 给不懂代码的家长/老师看的使用指南

> 这份文档是为"不会写代码、但想亲手验证这个工具靠不靠谱"的人准备的。
> 全程不需要打开任何代码文件，在 **Trae Solo** 的对话框里说中文就行。

---

## 一、这东西到底是干嘛的？

一句话：**它是高考志愿填报的"守门员"**。

想象一下，全国有 2000 多所大学、几万个专业。你孩子考完高考后，真正"能填"的只是其中一部分——有的是因为分数不够，有的是因为选科没选对，有的是因为色盲不能报化学，还有的是因为政审不过关。

这个工具做的事情，就是把"**孩子根本不能报的专业**"先挡在门外，剩下一张干净的清单交给下游的其他 模块（评估专业好坏、排冲稳保等）。

它**不**告诉你"这个专业好不好"、"值不值得报"——那是后面的事。它只回答一个问题：**"这个孩子能不能合法填这个志愿？"**

---

## 二、凭什么相信它给的结果是准的？

这一节最重要。请耐心看完。

### 保证 1：它根本没有"预测"，只有"查证"

很多志愿填报软件让你觉得它很"智能"，其实背后是在用模型猜。**这个工具反过来——它拒绝猜。**

举三个例子：

| 情况 | 普通软件常见做法 | 这个工具的做法 |
|---|---|---|
| 某个新开的专业没有历史录取分数 | 用"同校均值"估一个分数线，假装告诉你"稳"/"保" | **标记"历史数据缺失"，老实告诉你不知道** |
| 你没填孩子的位次 | 用分数换算一个位次继续跑 | **直接停下，要求你去查一分一段表** |
| 招生计划里写着"物理or化学" | 猜是"物理或化学" | **直接报错**，因为"or"可能是中文"或"的误写，也可能不是，不能猜 |

这就是为什么你看到的结果里常常有 `warnings`（提醒）和 `excluded_summary`（被排除的名单）——每一条都交代得清清楚楚：**为什么放进来、为什么踢出去**。没有一条是"模型觉得应该这样"。

### 保证 2：规则全部写在明面上，不藏在"AI 脑子"里

所有的过滤规则都写在三份 JSON 文件里（你不需要打开它们，只需要知道它们存在）：

- **`data/province_exam_mode.json`** — 哪个省是新高考、哪个省是传统文理
- **`data/physical_restrictions.json`** — 色盲不能报哪些专业、身高要求多少等等，**每一条都附带"来源文件"**（比如教育部 2003 年 3 号文件第二条）
- **`data/subject_requirement_keywords.json`** — 选科要求怎么解析

如果某天教育部发了新文件，改这几个 JSON 文件就行，**不用改代码**。这意味着：
1. 规则是可审计的——你可以让任何懂行的老师翻开这些文件逐条对照
2. 规则不会被"模型幻觉"篡改——它不是 AI 脑补出来的

### 保证 3：数据版本不对就停下

招生计划每年都变。这个工具在加载数据的时候会**强制检查版本**：
- 考生是 2026 年高考，但数据文件是 2024 年的 → **立刻停下报错**，绝不"凑合跑"

为什么这很重要？因为"用旧数据给新考生做决策"是志愿填报事故的重灾区。这个工具宁可不跑，也不骗你。

### 保证 4：任何异常都会大声告诉你

每次运行完，你会看到两个清单：
- `eligible_pool`：被留下来的院校和专业（基本盘）
- `excluded_summary`：被排除的院校和专业 + **为什么排除**

被排除的每一条都带原因。比如：
> `"target": "中国石油大学 · 化学工程与工艺", "reason": "体检维度：色盲考生不可报考化学类专业"`

如果你不同意，可以直接用这个原因去找招生老师核实。不会出现"我也不知道为什么它没推荐给我"这种黑箱情况。

## 三、在 Trae Solo 里怎么自然语言测试？

### 前提条件

- 你已经在 Trae Solo 里打开了这个项目（`/Users/liangchao/Downloads/jinli`）
- 左侧能看到 `.trae/skills/constraint-filter/` 这个文件夹

**不需要**装 Python、不需要写命令、不需要改代码。

### 准备材料：一份"考生信息"

随便找一张纸（或者一张电子便签）写清楚：

1. **省份**：比如 河南
2. **总分**：比如 580
3. **省内位次**（一分一段表查到的）：比如 45000
4. **科类**：物理类 / 历史类 / 理科 / 文科 / 综合改革
5. **选了哪 3 科**：比如 物理、化学、生物
6. **性别**：男 / 女
7. **体检情况**：色觉正常/色弱/色盲，视力多少，身高多少，听力是否正常
8. **政审是否清白**：是 / 否 / 不知道

就这些。没有任何"你不该填的东西"。

### 基础测试：来一段正常的

在 Trae Solo 的对话框里直接说：

> **"帮我用 constraint-filter 这个 skill 过滤一遍。考生信息：河南物理类考生，580分，位次 45000，选了物理化学生物，男生，色觉视力听力正常，身高 175，政审清白，高考年份 2026。"**

你应该会看到：
- 一份院校 + 专业的清单（eligible_pool）
- 每个专业上标了"冲/稳/保"
- 被排除的专业清单（excluded_summary），附带原因
- 警告清单（warnings，如果有缺字段）
- 统计信息（总共多少所、过滤后多少所）

### 进阶测试：故意"挑刺"——这才是验证它靠不靠谱的关键

> **"自动化工具最怕的不是正常输入，而是异常输入。一个不肯在异常时报错的工具，才是最危险的。"**

以下每一条都是你**应该**亲自试一遍的。看看它有没有老实地停下来。

#### 测试 1：故意不填位次

> "用 constraint-filter 过滤，河南物理类，580分，没填位次，选了物理化学生物，男生。"

**期望结果**：它应该**停下并报错**，提示 `INPUT_MISSING_REQUIRED_FIELD`，告诉你"位次比分数更重要，必须补齐"。

❌ 如果它用分数估算位次继续跑了——这个工具就不合格。

#### 测试 2：故意写一个色盲考生报化学

> "用 constraint-filter 过滤，河南物理类，580分，位次 45000，选了物理化学生物，男生，**色盲**，其它正常。"

**期望结果**：运行成功，但被排除的名单里应该出现"化学工程与工艺"这样的专业，原因写明"色盲考生不可报考化学类专业"，并标注依据是教育部哪一条文件。

#### 测试 3：故意选一个新高考 3+1+2 省份同时选物理和历史

> "河南考生（3+1+2），物理类，选了物理、历史、化学。"

**期望结果**：它必须报 `INPUT_INVALID_SUBJECT_COMBINATION`，因为物理和历史互斥。

#### 测试 4：故意填一个不存在的省份

> "火星省考生，物理类，580 分……"

**期望结果**：报 `INPUT_INVALID_FIELD_VALUE`，告诉你省份不在 31 个省列表里。

#### 测试 5：故意把高考年份写成 2027

> "……高考年份 2027。"

**期望结果**：报 `DATA_SOURCE_UNAVAILABLE` 或 `DATA_VERSION_MISMATCH`，明确告诉你数据版本不对，拒绝跑。

#### 测试 6：政审不知道

> "……政审情况不知道（null）。"

**期望结果**：运行成功，但 warnings 里应该有一条"政审维度未核实，涉及公安/军事/国防等专业已保留在池中，请自行核对"；公安大学等学校仍然保留在池里。

## 四、读结果时的三个小窍门

1. **先看 `status`**：`"success"` 才算跑通，`"error"` 就是出事了，看 `user_action_required` 字段——它会用中文告诉你怎么办
2. **`warnings` 不可忽略**：哪怕 status 是 success，warnings 里的每一条都值得读——那是工具主动告诉你"这部分我没办法帮你过滤，你自己当心"
3. **`excluded_summary` 逐条核对**：如果你觉得某个本该能报的专业被排除了，直接把 `target` 和 `reason` 发给高中老师或招办问一下。原因是明示的，方便追溯

## 五、经常出现的疑问

**Q1：这工具会不会把我孩子本可以报的专业漏掉？**

可能会发生。这是**有意的保守设计**——宁可少推荐，也不要误导你去填一个孩子根本不能报的志愿。少推荐是"少了个选项"，误导是"白白丢掉一年"。被排除的专业都有明确原因，你可以逐条复核；如果你觉得某条不对，告诉维护者改 JSON 规则即可，不用动代码。

**Q2：为什么它不直接帮我选出最好的志愿？**

因为这个 skill 只是整个系统的**第一关**（硬约束过滤）。它后面还有 4 个 模块 负责评估专业价值、匹配院校、排冲稳保、做特殊适配。把决策职责拆开，是为了每一步都能单独审计，而不是让你面对一个"看起来很专业但说不清为什么"的黑箱推荐。

**Q3：数据是哪来的？真的可信吗？**

当前 MVP 阶段的数据是**示范数据**（`data/admission_plans/henan_2026.json`），只包含少数几所学校，用于演示流程。真正上线前，必须由维护者从河南省招生办公室官方发布的《普通高校招生计划》里整理入库。换句话说：
- 工具的**逻辑**是已经准备好的（你可以用示范数据测）
- 工具的**数据**需要每年替换成官方权威版本（这不是代码能解决的事，需要人工对照）

你在 `data/admission_plans/henan_2026.json` 的开头能看到 `"source"` 字段——那就是数据来源的说明，每次换数据都应该更新这个字段。

**Q4：我发现某条规则不对，怎么反馈？**

把被排除/保留的具体条目（`target` + `reason`）以及你认为正确的答案告诉维护者即可。维护者会修改 `data/` 下相应的 JSON 文件，**不需要改代码**。下次运行时新规则自动生效。

**Q5：能不能让它告诉我"大概率能上清北"？**

**不能，也不会。** 这个工具只做"能不能合法填报"的判断，不对未来分数线做任何预测——因为任何预测都是赌博。关于"冲、稳、保"的档位判定用的是过去 3 年的真实位次加权平均，不是模型预测。
