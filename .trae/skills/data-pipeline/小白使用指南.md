# 给不懂代码的家长/老师看的使用指南

> 这份文档是为"手里有招生计划PDF、但不知道怎么让系统用上它"的人准备的。
> 全程不需要打开任何代码文件，在 **Trae Solo** 的对话框里说中文就行。

---

## 一、这东西到底是干嘛的？

一句话：**它是数据工厂——把你手里的官方文件变成系统能读懂的格式。**

整个志愿填报系统分两步走：

```
你手里的官方文件（PDF / Excel / Word）
        ↓ 数据准备 做这一步
干净的 JSON 数据
        ↓ constraint-filter 做这一步
"孩子能报哪些专业"的过滤结果
```

没有 数据准备 喂进去的数据，constraint-filter 就无从过滤。所以 数据准备 是整个流程的**最前端**，它的工作就一件事：**原封不动地把官方文件的内容搬进系统，不猜、不改、不补**。

它支持三类文件，对应三类数据：

| 你手里的文件 | 数据类型 | 何时用 |
|---|---|---|
| 各省招生计划（大厚本）| 招生计划 | 每年 5~6 月出分前导入 |
| 近 3 年各校专业最低位次 | 历史录取位次 | 录取结束后（7~8 月）导入 |
| 省内一分一段表 | 一分一段表 | 出分当天导入 |
| 单所学校招生章程 | 招生章程 | 需要核对单科要求/身高视力时 |

## 二、凭什么相信它处理的结果是准的？

### 保证 1：它不改原文，一个字不动

招生计划里的选科要求写的是"物理(1科必选)且(化学or生物选1)"，系统就照着原文存进去，一个字不改。哪怕这句话有歧义，系统也不猜——它把原文交给下游的 constraint-filter 去解析，解析不了就报错，不会悄悄给你一个错误的结论。

你可以随时打开写进去的 JSON 文件，第一眼就能看到 `"subject_requirement_raw"` 字段的内容和原文件一模一样。

### 保证 2：找不到的字段，留空不估

举个例子：学校代码本来应该是教育部统一的 5 位数字，但如果某份文件里有学校没写代码，系统会：

- 留 `null`
- 在校验报告里用 `⚠️` 标出来告诉你

**不会**用别的学校的代码顶替，也不会随便填一个。

### 保证 3：每次导入都要你眼睛过一遍

数据处理分两个阶段，中间有一个**强制确认**：

1. 系统先解析、归一化、给你看校验报告（有多少所学校、有哪些字段有问题）
2. **你确认之后**，数据才真正写入系统

校验报告里有 ❌（必须修复）/ ⚠️（注意）/ ✅（通过）三档，不会悄悄跳过问题。

### 保证 4：数据来源永远可追溯

写进去的每份数据都带着：
- `"source_raw_filename"`：原始文件名
- `"fetched_at"`：入库日期
- `"source"`：用户说明的来源
- `"source_type": "user_upload"`：标明是用户上传，未经官方数据库交叉验证

constraint-filter 出结果时会把这个来源标在结果里，让你随时知道这批数据的出处。

## 三、在 Trae Solo 里怎么用？

### 前提条件

- 你已经在 Trae Solo 里打开了这个项目（`/Users/liangchao/Downloads/jinli`）
- 手里有一份官方文件，知道它存在电脑上的完整路径（比如 `/Users/你的名字/Downloads/henan_2026.pdf`）

**不需要**装 Python、不需要写命令、不需要改代码。

---

### 标准流程：导入一份招生计划

#### 第一步：告诉系统你有什么文件

在对话框里说：

> "我有一份河南 2026 年招生计划，文件在 `/Users/你的名字/Downloads/henan_2026.pdf`，帮我导入。"

系统会先**识别文件**，告诉你它认为这是什么类型的文件，问你对不对。

#### 第二步：看系统给你的校验报告

系统提取和归一化完成后，会打印一份报告，大概长这样：

```
📋 解析摘要
============================================================
  省份：河南  年份：2026
  学校：312 所  专业组：1842 个  专业：6108 个
  来源文件：henan_2026.pdf

⚠️  注意事项：
   · 23 个专业的 school_code 不是 5 位数字，请核对
   · 4 个专业的 subject_requirement_raw 含有特殊字符，请人工确认

✅ 验证通过
```

看一遍报告，有 ❌ 就先让维护者处理；只有 ⚠️ 可以先跑，但记住要事后人工核对那几条。

#### 第三步：确认，让系统写进去

你觉得没问题（或可以接受 ⚠️ 的瑕疵），就说：

> "好的，写进去吧。"

系统写入成功后会告诉你：

- 文件存到了哪里（完整路径）
- 包含多少所学校、多少个专业
- 有哪些字段仍为 null，需要手工补全
- 一条提醒："此数据来自您上传的文件，请自行核对关键专业的信息"

## 四、几个常见场景怎么说

#### 场景 1：我有招生计划，但不确定格式对不对

> "帮我识别一下这个文件是什么类型的：`/Users/我/Downloads/招生计划.xlsx`"

系统会告诉你它识别出来的文件类型（PDF/Excel/Word）和数据类型，不确定的会直接问你。

---

#### 场景 2：我想同时导入招生计划和历史位次

两个文件分两次说：

> "先帮我导入招生计划：`/路径/henan_2026.pdf`，省份河南，年份 2026。"

确认写入后再说：

> "再帮我导入历史录取位次：`/路径/henan_history_ranks.xlsx`，也是河南 2026。"

系统会分别处理、分别写入，两个文件互不影响——这就是离散存储的意义：改一个不影响另一个。

---

#### 场景 3：我不知道文件路径

在 Mac 上：打开 Finder，找到文件，按住 Option 键并右键点击文件，选"拷贝路径"，粘贴过来就行。

---

#### 场景 4：系统说"无法解析此文件"

常见原因和解决方法：

| 原因 | 怎么办 |
|---|---|
| 加密 PDF（需要密码才能打开）| 先解锁（用 Adobe Acrobat 或在线工具），再导入 |
| 扫描件（图片型 PDF，不含文字）| 向省考试院要 Excel 版本，或者找带文字层的 PDF |
| 格式太奇特（多层嵌套合并单元格）| 先手工整理成标准表格，再导入；或告知维护者 |

## 五、验证它有没有做错事

### 测试 1：故意给一个你知道内容的小文件

手边有一份只有 3~5 所学校的示范 Excel（或者你自己整理一个），让系统导入。导入后打开 JSON 文件，逐条核对：
- 学校名对不对
- 选科要求原文有没有被改动
- 计划招生人数对不对

这是最直接的核对方式。

### 测试 2：故意给一个年份对不上的文件

文件里写的是 2024 年，但你告诉系统年份是 2026：

> "导入这份文件，省份河南，年份 **2026**。"（但文件实际是 2024 年的）

**期望结果**：校验报告里出现 ❌ 或 ⚠️，提示年份与文件内容不符，需要你确认。不应该悄悄用错误的年份写进去。

### 测试 3：故意给一个空文件或损坏文件

**期望结果**：系统报告"提取失败，原因：文件内容为空/格式不支持"，不继续后续步骤，不写入任何东西。

## 六、和 constraint-filter 是什么关系？

两个工具是**上下游关系**，各管一段：

```
数据准备              constraint-filter
（数据工厂）               （过滤机器）
    ↓                           ↓
把文件变成 JSON      →    拿 JSON 给考生过滤志愿
    ↓                           ↓
管"数据从哪来"           管"考生能报哪些"
管"数据准不准"           管"过滤规则对不对"
```

数据准备 出了问题（数据不对）→ constraint-filter 的结果再漂亮也没用。  
所以**导入完数据，第一件事是用 constraint-filter 跑一遍看看结果是否合理**，而不是直接相信存进去的数字。

## 七、经常出现的疑问

**Q1：导入会覆盖旧数据吗？**

不会自动覆盖。如果目标路径已有同名文件，系统会停下来告诉你，让你决定是否覆盖。你可以在命令里加 `--overwrite` 选项，但系统不会在你没说的情况下自己覆盖。

**Q2：我手里的文件是从网上下的，不是官方来的，能用吗？**

能用，但系统会在结果里标注 `source_type: user_upload`，提醒你这批数据没经过官方渠道交叉验证。建议关键专业（尤其是特殊类型招生、定向生）务必手工核对一遍。

**Q3：提取出来的数据有错字，要怎么改？**

两种办法：
1. **改原文件**（推荐）：在官方发布的 Excel 里更正，然后重新导入
2. **改 JSON 文件**（临时应急）：直接打开写入的 JSON 文件，找到那一条手工修改，但下次重新导入会被覆盖

**Q4：为什么 `yearly_admission_data` 导入后是空的？**

因为招生计划（5~6 月发布）和历史录取位次（7~8 月发布）是**两份独立的文件**，导入时间不同。招生计划只管"能不能报"，历史位次管"大概率录多少分"。两份数据要分别导入，constraint-filter 会在运行时自动把它们拼起来。

**Q5：一分一段表有什么用？**

目前用于辅助核实：考生填的省内位次有没有填错（比如和一分一段表对不上）。这个功能在 MVP 阶段是可选项，即使没导入也不影响主流程。
